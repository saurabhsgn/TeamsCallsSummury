## Summary

The meeting focused on how to validate and deploy the routing simulator so the team can compare dispatcher-created routes with what the optimizer would have produced under the same timing conditions. The discussion covered forecasting and SLA data needed to support the model, the architectural constraints for running the simulator in production safely, and the work required to finalize the optimizer functional spec and supporting rules before Q2.

## Decisions

- The simulator should ultimately compare dispatcher route choices against optimizer output based on the `build route` event and available order-integration timing.
- For production-oriented simulation, the flow needs to be event-based rather than only a standalone UI test tool.
- The simulator logic must not run against the real-time production database in a way that affects performance.
- The optimizer should consider the broader eligible invoice pool rather than only the exact invoices selected manually by the dispatcher.
- Average picking-time buffers are required as assumptions for simulation, and they should be tied to SLA-driven scenarios such as `platinum`, `silver`, and similar order classes.
- The first Q2 production priority is the practical simulation or optimized-input path needed for rollout; the more future-looking continuous listener model is a later step.
- Zip-code-based grouping can be tried as one optimization rule, but it should be discarded if it does not improve route quality.
- The optimizer functional spec needs to be completed and signed off by next Thursday.

## Open Items

- The team still needs the final production design for capturing simulation data without writing to the real-time DB or hurting performance.
- It is still unclear what the best lower-environment or QA approach is for replaying realistic production routing behavior.
- The team needs confirmed average picking-time assumptions by order or SLA type.
- The exact boundary between what is needed immediately for Q2 and what belongs to the longer-term automated listener model still needs to stay clearly separated.
